# チケット振り返り — persona: DEV（定義は `start-ticket/SKILL.md`）

チケット1件の実行経験を学習データとして、プロセス・スキルの改善に蒸留する。**提案は無条件に取り込まず、人が改善を見込めると判定したものだけ反映する**（過適合の防止）。

## 起動条件

対象チケットが `done`（またはPR作成済みで人が明示指示した場合）。

## 原則

1. **事実はファイルから**: トラジェクトリは指示書（§4.2・§4.3）・質疑・レビュー報告（BJ指摘と対応欄）・git log（`T-NNN`）から再構成する。記憶や自己評価で書かない。DEV自身の失敗もBJの指摘記録を根拠にそのまま書く。
2. **ループを継続する**: `.docs/note/` の直近の振り返りを読み、そこで定義された**検証信号を今回の実測値で更新**する。前回「再発時に採用」とされた提案が再発していれば、今回は採用推奨に昇格させる。
3. **N=1で書き換えすぎない**: 即時採用を推奨してよいのは「機構的な穴」（チェック手順の欠落・検出クラスの不足など、再発が構造的に確実なもの）のみ。単発の可能性があるものは「再発時に採用」とする。
4. **編集提案は具体的に**: 対象ファイルと追記・変更する文言（1行〜数行）まで書く。「気をつける」で終わらせない。

## 出力（`.docs/note/retro-YYYYMMDD-TNNN.md`）

| セクション | 内容 |
|---|---|
| 1. トラジェクトリ | 局面×起きたこと×分類（正常／関門が機能／弱点）。関門の停止・BJ指摘・QAラウンド・例外対応を漏らさない |
| 2. 検証信号 | 前回振り返りの指標を引き継ぎ実測（前回値→今回値）。必要なら指標の追加・廃止を提案 |
| 3. 教訓→編集提案 | 表形式: 教訓／編集提案（対象ファイル・文言）／推奨（採用推奨・再発時に採用・keep・採用済み） |
| 4. うまく機能したもの | 変更しないと明記する（守るべき設計を提案の巻き添えにしない） |

## 停止

「振り返りを作成しました。提案の採否を指示してください（例:『AとCを採用』）。採用分のみ対象ファイルへ反映し、振り返りに採否結果を追記します」

## 採否後の反映

- 人が採用を指示した提案のみ、対象ファイルへ最小差分で反映する。
- 振り返りの提案表に採否結果（採用日・反映先）を追記する。
- スキル・プロセス文書（Git管理）の変更はPRレビューの対象。カタログに波及する提案が出た場合はチケット化する（上流優先）。
