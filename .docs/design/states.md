# states — 状態定義

> コンテキストは情報定義（information.md）と名前を揃える。状態一覧は遷移形式（1行=1遷移）で、**遷移は各状態モデル内で閉じる**（他モデルへの波及は説明列に記す）。遷移UCは buc.md で発番済みの UC を参照する。
> **終端の表記**: 遷移先状態「—」はそのモデルの終端（以降の遷移なし）。図では `[*]` への辺として表し、遷移UCがあればラベルに記す。

## 状態モデル

| ID | コンテキスト | 状態モデル | 対象情報 |
|---|---|---|---|
| STM-01 | アカウント | アカウント状態 | INF-01（ユーザー情報） |
| STM-02 | セッション | セッション状態 | INF-03（アクセストークン）・INF-04（リフレッシュトークン） |
| STM-03 | ワンタイムトークン | パスワードリセット状態 | INF-05（パスワードリセットトークン） |

## 状態遷移図（PlantUML・状態モデル別）

### STM-01 アカウント状態

```plantuml
@startuml
state "メール未確認" as mail_unverified
state "招待済み未受付" as invited
state "未認証" as inactive
state "無効化済み" as disabled
state "削除済み" as deleted

[*] --> mail_unverified : UC-002 アカウント登録完了
[*] --> invited : UC-011 管理者招待完了
mail_unverified --> inactive : UC-003 メール確認完了
mail_unverified --> mail_unverified : UC-004 再送信（確認トークン期限切れ）
invited --> inactive : UC-012 招待受付完了
inactive --> disabled : UC-014 無効化
inactive --> deleted : UC-016 削除
disabled --> inactive : UC-015 再有効化
deleted --> [*]
note right of inactive
  未認証=ログイン可能。
  UC-005 ログイン成功で
  セッション（STM-02）が生成される
end note
@enduml
```

### STM-02 セッション状態

```plantuml
@startuml
state "認証済み" as active
state "トークン期限切れ" as at_expired
state "セッション失効" as revoked

[*] --> active : UC-005 ログイン成功
active --> at_expired : アクセストークン有効期限到達（VAR-03）
active --> revoked : UC-013 / UC-010 / UC-017 / UC-009 / 再利用検知（VAR-10）
active --> revoked : UC-014 / UC-016（アカウント側の無効化・削除に伴う）
active --> revoked : UC-007 ログアウト
at_expired --> active : UC-006 トークン再発行成功
at_expired --> revoked : リフレッシュトークン期限切れ（VAR-04） / UC-013
revoked --> [*]
note right of revoked
  クライアントが401検知し
  ローカルセッションをクリア。
  再ログイン（UC-005）は新セッション
end note
@enduml
```

### STM-03 パスワードリセット状態

```plantuml
@startuml
state "パスワードリセット中" as resetting

resetting --> [*] : UC-009 リセット完了（トークン使用済み。全セッション無効化へ波及）
resetting --> [*] : リセットトークン期限切れ（VAR-05）
@enduml
```
