# non-functional-requirements — 非機能要求定義

| ID | 分類 | 要求 |
|---|---|---|
| NFR-01 | セキュリティ | パスワードはbcryptでハッシュ化すること（コストファクター: 12。OWASP 2023推奨値。Go実装では `bcrypt.GenerateFromPassword(password, 12)` を使用すること。bcryptを採用した理由: Go標準ライブラリへの組み込みが容易で実績があり、POCスコープとして適切。プロダクション移行時はargon2id（RFC 9106）への移行を検討すること） |
| NFR-02 | セキュリティ | JWTの署名アルゴリズムはRS256を使用すること（auth-serviceが秘密鍵で署名、各app-serviceは公開鍵で検証） |
| NFR-03 | セキュリティ | ログイン失敗10回で15分ロックアウトするレートリミットを設けること。失敗回数（INF-08）・ロック状態（INF-09）はRedis（EXT-02）のTTL付きキーで管理し、15分後に自動解除すること。ロックアウト中のログイン試行には429を返し、RFC 9457準拠レスポンスに`error_code: account_locked`（VAR-11）と`retry_after`（解除時刻）を含めること。ユーザー不存在時はtiming attack対策のためbcryptダミー検証を実行すること。失敗カウントはユーザーの存在有無に関わらず同一メールアドレスに対して均一に加算すること（ロックアウト動作差異によるユーザー列挙防止） |
| NFR-04 | 可用性 | ヘルスチェックAPI（BUC-S01）で死活監視できること |
| NFR-05 | 保守性 | auth-serviceとapp-serviceはモノレポで独立したサービスとして管理すること |
| NFR-06 | 仕様 | エラーレスポンスはRFC 9457（application/problem+json）に準拠すること。セッション失効時は401レスポンスの`revocation_reason`拡張フィールドに失効理由コード（VAR-10）を含めること。メール未確認アカウントへのログイン試行には403で`type: .../email-not-verified`を返すこと。無効化済みアカウントへのログイン試行には403で`type: .../account-disabled`を返すこと（いずれもパスワード検証成功後に返す） |
| NFR-07 | 監査ログ | 下表の操作を監査ログとして記録すること。ログはLogging Guideline v1.0に準拠した構造化JSONとし、Go向けに実装すること（slogまたはzap）。インフラ連携（CloudWatch等）はスコープ外とし、ローカル開発ではstdoutへ出力すること |
| NFR-08 | ログレベル | Logging Guideline v1.0のログレベル定義に従うこと。UseCase開始/終了はINFO、ビジネス例外はWARNING、外部依存失敗はERROR、未捕捉例外はCRITICAL |
| NFR-09 | ログフィールド | 全ログに`ts`（ISO-8601 UTC）、`lvl`、`svc`、`ctx`、`trace_id`/`span_id`（OpenTelemetry）、`req_id`、`msg`を含めること。機密情報（パスワード・トークン・個人情報）はログに含めないこと |
| NFR-10 | セキュリティ | リフレッシュトークン（INF-04）はローテーションすること。使用のたびに新トークンを発行し古いトークンを無効化する。古いトークンの再利用を検知した場合は全セッションを無効化し再ログインを要求すること。実装方式はNFR-14参照 |
| NFR-11 | セキュリティ | パスワードリセット要求に対してレートリミット（VAR-12）を設けること。同一メールアドレスへの送信は5分に1回に制限しRedis（EXT-02）で管理すること |
| NFR-12 | セキュリティ | メール確認トークン再送信に対してレートリミット（VAR-13）を設けること。同一メールアドレスへの送信は5分に1回に制限しRedis（EXT-02）で管理すること |
| NFR-13 | セキュリティ | 招待トークン送信に対してレートリミット（VAR-14）を設けること。同一メールアドレスへの送信は5分に1回に制限しRedis（EXT-02）で管理すること |
| NFR-14 | セキュリティ | （NFR-10実装要件）リフレッシュトークン（INF-04）はopaque tokenとして実装すること。`crypto/rand` で32バイトのランダム値を生成しbase64urlエンコードしたものをクライアントに返し、DBにはSHA-256ハッシュ値のみを保存すること。平文トークンを永続化してはならない。各レコードは `token_id`（UUID、DB主キー）・`family_id`（同一ローテーションチェーンの識別子、UUID）・`parent_token_id`（前トークンの参照）を持ち、再利用検知は同一 `family_id` を持つ全レコードを一括失効させることで実装すること |
| NFR-15 | セキュリティ | ワンタイムトークン（INF-05 パスワードリセット・INF-06 メール確認・INF-07 招待）はopaque tokenとして実装すること。`crypto/rand` で32バイトのランダム値を生成しbase64urlエンコードしたものをメールのURLパラメータのみに含め、DBにはSHA-256ハッシュ値のみを保存すること。平文トークンを永続化してはならない |
| NFR-16 | 仕様 | JWTの `roles` クレームにユーザーのロール（INF-02）を文字列配列として格納すること（例: `["user", "operator"]`） |

## 監査ログ対象操作（NFR-07）

| 操作 | ログレベル |
|---|---|
| ログイン成功 | INFO |
| ログイン失敗 | WARNING |
| パスワードリセット | INFO |
| パスワード変更 | INFO |
| トークン強制失効 | INFO |
| 管理者招待 | INFO |
| アカウント無効化 | INFO |
| アカウント再有効化 | INFO |
| アカウント削除 | INFO |
| ロール変更 | INFO |
| 不正トークンでのアクセス試行 | WARNING |
| リフレッシュトークン再利用検知（全セッション無効化） | CRITICAL |
| トークン再発行成功 | INFO |
| JWT署名不正・改ざん検知 | WARNING |
| JWTフォーマット不正 | WARNING |
